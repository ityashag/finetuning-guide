The Why Behind Fine-Tuning

A first-principles companion to the Complete Guide. Every formula explained term by term. Every hyperparameter dissected. For each thing: what problem it solves, and what breaks if you change it.

How to read this

Every concept follows the same pattern: a plain-English explanation first, then the formula, then a box explaining what every symbol means, then a box about why this particular form was chosen, then a box about what happens if you change parts of it. If you only read the colored boxes, you'll still understand 80% of fine-tuning.

Chapter 0

0. The absolute basics — what is a "parameter," "training," "loss"?

Before any formula, let's name the pieces, because every page of this document will use these terms.

0.1 What is a parameter?

A neural network is just a giant function. You give it an input, it gives you an output. The function has billions of knobs inside it — numbers that determine what it does. Each knob is a parameter. A "7B model" has 7 billion such knobs. A "70B model" has 70 billion.

Each parameter is just a single floating-point number, like 0.0312 or -1.7724. The collection of all these numbers is what gets saved to disk when you "download a model." The Llama-3-8B weights file is about 16 GB — that's 8 billion numbers stored as 16-bit floats.

A pretrained LLM is like an enormous mixing board with 8 billion sliders. Each slider has been set, very carefully, during pretraining (which cost millions of dollars and burned terabytes of text data). Inference is pushing audio through the mixing board with the sliders fixed. Fine-tuning is going in and turning the sliders.

0.2 What is training? What is a "loss"?

Training means: repeatedly adjust the knobs to make the model less wrong. You need three things:

  1. A dataset of (input, desired-output) pairs. Example: input = "Translate to French: Hello", desired output = "Bonjour".
  2. A loss function $\mathcal{L}$ — a number that says "how wrong was the model on this example?" Smaller loss = closer to the truth.
  3. An optimizer (usually AdamW) that uses calculus to figure out which direction to nudge each knob to make the loss smaller.

For language models, the loss is almost always cross-entropy on the next token:

$$\mathcal{L} = -\sum_{t} \log P_\theta(\text{correct token}_t \mid \text{previous tokens})$$
$P_\theta$ is "the probability the model with parameters $\theta$ assigns." $-\log P$ is large when the model gave the correct token a small probability, and tiny when the model assigned high probability to the right answer. Sum over all positions in the sequence and you have one number to minimize.
Why log? Two reasons. (1) Mathematically, log-probability adds across independent tokens, while raw probability multiplies and rapidly underflows to zero for long sequences. (2) Log makes the gradient nicely behaved — the gradient of $-\log P$ is large when $P$ is small (so the model gets a big nudge when it's very wrong) and tiny when $P$ is close to 1 (small nudge when nearly right).
If you used $-P$ instead of $-\log P$: the loss would still be smaller for higher $P$, but a model that's 99% confident and one that's 99.99% confident would get nearly identical gradients. With $-\log P$, the latter case gives a much smaller gradient, which is correct — you don't need to push hard when the model is already nearly right.

0.3 What is "gradient descent"?

Calculus tells you, for any knob $\theta_i$, what direction nudging it would move the loss. That signed nudge is the gradient, written $\frac{\partial \mathcal{L}}{\partial \theta_i}$. The update rule (simplified) is:

$$\theta_i \leftarrow \theta_i - \eta \cdot \frac{\partial \mathcal{L}}{\partial \theta_i}$$
$\eta$ (Greek letter "eta") is the learning rate. It's the step size. The minus sign is because gradient points uphill on loss and you want to walk downhill. Do this trillion times per parameter and you've trained a model.
A learning rate exists because the gradient only tells you a direction, not how far to move. Move too far and you overshoot the valley; move too little and you'll never get there.
If $\eta$ is too large: loss spikes, training diverges (you see NaN losses). If too small: the loss decreases but achingly slowly, and you may stop training before convergence. Picking $\eta$ is the single most consequential hyperparameter choice. We will come back to it many times.

0.4 What is "fine-tuning" vs. "pretraining"?

Pretraining: start from totally random knob positions, train on trillions of tokens of internet text with next-token prediction. Outcome: a model that knows grammar, facts, basic reasoning, code — a general-purpose engine. Costs millions of dollars.

Fine-tuning: start from a pretrained model. Continue training, but on a small task-specific dataset, usually for far fewer steps. Outcome: a model that's better at your task — answering customer questions in your company's style, doing legal summarization, writing code in a particular framework. Costs hundreds of dollars to thousands.

Pretraining is too expensive to repeat. Fine-tuning lets us reuse the expensive pretrained knob positions as a starting point and only do the cheap last-mile adjustment for our task. The cleverness of LoRA, QLoRA, GRPO, etc. is all about making fine-tuning even cheaper and more reliable.
Chapter 1

1. From words to vectors — what an "embedding" actually is

You can't do calculus on the word "cat." You can do calculus on numbers. So step one of every LLM is turning text into numbers.

1.1 Tokens and vocabulary

Text is first split into tokens by a tokenizer. Tokens are usually subword chunks: "unhappiness" becomes ["un", "happiness"] or ["un", "happy", "ness"] depending on the tokenizer. A model has a fixed vocabulary — Llama 3's is 128,256 tokens. Every input token is just an integer ID from 0 to 128,255.

1.2 The embedding table

The first layer of a transformer is a giant lookup table: embedding matrix $E \in \mathbb{R}^{V \times d_{\text{model}}}$, where $V$ is vocabulary size and $d_{\text{model}}$ is the model's hidden dimension (e.g., 4096). Row $i$ is the vector for token $i$.

$$x_t = E[\text{token\_id}_t]$$
$x_t \in \mathbb{R}^{4096}$ is a vector of 4096 numbers. Every token in the vocabulary has a learned 4096-dimensional vector. These vectors are learned during pretraining — they aren't hand-designed.
Why high-dimensional? Because language has many independent dimensions of meaning — sentiment, formality, topic, plurality, tense, register, named-entity-or-not, code-or-prose, and thousands more. One dimension cannot capture all of these; you need a wide vector where different directions encode different aspects.
A famous result with old word embeddings was: vector("king") − vector("man") + vector("woman") ≈ vector("queen"). The vectors organize themselves into geometric relationships that reflect meaning. Modern transformer embeddings are richer but the principle is the same: similar meanings → nearby vectors.
Chapter 2

2. Attention — the formula, term by term, fear-free

2.1 The problem attention solves

You're reading: "The lawyer who Maria hired won the case because she had prepared thoroughly." Who is "she"? You scan back through the sentence, ask "what could 'she' refer to?", weigh "lawyer" against "Maria", and pick. That weighing of distant words is the operation attention performs, but inside the neural network, in parallel, for every token at once.

Before attention, recurrent networks (RNNs) processed text left-to-right and squeezed history into a single hidden state. Distant context degraded. Attention is a way to let every token directly consult every other token in one step, with learned weights.

2.2 Three roles per token: Query, Key, Value

For each token, the model produces three different vectors from the same input:

RoleMental modelUsed for
Query (Q)"What information am I looking for?"This token's own search request
Key (K)"What information do I advertise?"What another token offers when matched
Value (V)"What payload do I carry?"The content actually mixed in
Imagine a room of 10 people. Each holds up two signs: one says "I'm asking about X" (their query), the other says "I know about Y" (their key). They look around, see whose key matches their query, and copy notes from those people (the value). Attention does this with learned vectors instead of text labels.
Why three separate vectors instead of one? Because what you need to find and what you advertise are not the same. A pronoun "she" needs to find an antecedent (query: "looking for a female noun nearby"). The word "lawyer" advertises "I am a profession, animate, can be female" (key). The actual content mixed into "she"'s representation — like which lawyer specifically, including her actions — is different again (value). Separating Q, K, V lets the model learn these three roles independently.

2.3 Producing Q, K, V

You take the input embeddings $X \in \mathbb{R}^{n \times d}$ (a stack of $n$ token vectors, each of dimension $d$). You apply three learned weight matrices:

$$Q = X W_Q, \quad K = X W_K, \quad V = X W_V$$
$W_Q, W_K, W_V \in \mathbb{R}^{d \times d_k}$ are three independent weight matrices. They're learned during pretraining. Each takes a $d$-dimensional input vector and produces a $d_k$-dimensional output vector. $Q$, $K$, $V$ are each matrices of shape $n \times d_k$ — one $d_k$-vector per token. These matrices, in HuggingFace's Llama code, are named q_proj, k_proj, v_proj. That's exactly where those names in target_modules come from.
If we used the same matrix for Q and K (i.e., $W_Q = W_K$): then a token's query and key would be identical, and the dot product $Q K^\top$ would just be self-similarities. The model would mostly attend to itself. Empirically: bad. Modern models always learn Q and K separately.

2.4 The dot-product score: $QK^\top$

Now we compute how much every query matches every key:

$$S = Q K^\top \in \mathbb{R}^{n \times n}$$
$S_{ij}$ is the dot product of query vector $i$ with key vector $j$. The dot product $\mathbf{q} \cdot \mathbf{k} = \sum_l q_l k_l$ is large when $\mathbf{q}$ and $\mathbf{k}$ point in similar directions, near zero when they're perpendicular, large-negative when opposite. So $S_{ij}$ measures "how relevant is token $j$ to token $i$'s query?"
Why dot product specifically? Because it's the simplest learnable similarity. The model's weights learn what makes two vectors "similar enough" for a strong attention link. Cosine similarity would work too but requires normalization; dot product is cheaper and just as effective once $W_Q, W_K$ have been trained.

2.5 The scale: $\sqrt{d_k}$

Divide $S$ by $\sqrt{d_k}$:

$$S' = \frac{QK^\top}{\sqrt{d_k}}$$
Why divide? Imagine $\mathbf{q}$ and $\mathbf{k}$ are each $d_k$-dimensional vectors whose entries are independent with variance 1. Their dot product $\sum_l q_l k_l$ is a sum of $d_k$ random terms each with variance 1, so the sum has variance $d_k$ and standard deviation $\sqrt{d_k}$. So dot products grow in magnitude with $d_k$. With $d_k = 128$, raw scores could be 10-20 in magnitude. With $d_k = 4096$ (no multi-head split), they'd be in the 50s. Dividing by $\sqrt{d_k}$ normalizes this back to standard deviation ~1.
What if we skipped the division? Without it, large $d_k$ values produce huge attention scores. After softmax, almost all probability piles onto a single token (because $e^{20}$ vs $e^{19}$ is a factor of $e$ — extreme winner-take-all). Then gradients through softmax become near-zero (saturation), and training stalls. The $\sqrt{d_k}$ scaling keeps softmax inputs in a reasonable range so gradients flow.
What if we divided by $d_k$ instead of $\sqrt{d_k}$? Scores would become too small, softmax would be too flat (close to uniform), and the model couldn't focus attention sharply. $\sqrt{d_k}$ is the Goldilocks rate determined by the variance argument above.

2.6 Softmax — turning scores into weights

Apply softmax along each row of $S'$:

$$A_{ij} = \frac{\exp(S'_{ij})}{\sum_{j'} \exp(S'_{ij'})}$$
For each row $i$ (token $i$'s view of the world), $\{A_{ij}\}_j$ is a probability distribution over all other tokens summing to 1. $A_{ij}$ is "the fraction of attention token $i$ pays to token $j$."
We need attention weights to (a) be non-negative — you can't pay -3% attention — and (b) sum to 1 — total attention is a budget. Softmax is the standard way to convert arbitrary real numbers into a probability distribution while preserving their order: larger inputs get larger outputs, and the conversion is differentiable everywhere.
If we just normalized (divided by sum without exponentiating): negative scores would become "negative attention," which is meaningless. If we used ReLU then normalized: it would work, but ReLU's hard zero means many tokens have exactly zero attention, and gradients can't flow through them — softmax is "soft" precisely because every token still gets some attention, however small, so gradients can adjust during training.

2.7 Mixing values: $AV$

Finally, use the weights to mix values:

$$\text{output} = A V$$
For token $i$, the output is $\sum_j A_{ij} V_j$ — a weighted average of all value vectors, with weights from token $i$'s attention distribution. Token $i$'s new representation has "absorbed" content from the tokens it attended to most.

2.8 The full formula, decoded

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right) V$$

Read it now in plain English: "Compute how much each query matches each key (dot product), shrink so softmax behaves (divide by $\sqrt{d_k}$), convert to probabilities (softmax), then mix values by those probabilities."

2.9 Multi-head attention — why multiple heads?

One attention computation captures one kind of relationship. Real language has many kinds happening at once: coreference, syntax, topic-tracking, register, etc. Solution: do attention multiple times in parallel with different learned weights, each capturing different relationships. Each parallel computation is a "head."

$$\text{MultiHead}(X) = \text{Concat}(\text{head}_1, \ldots, \text{head}_h) W_O$$ $$\text{head}_i = \text{Attention}(X W_Q^{(i)}, X W_K^{(i)}, X W_V^{(i)})$$
If $d_{\text{model}} = 4096$ and you use $h = 32$ heads, each head's $d_k = 4096/32 = 128$. Each head learns its own $W_Q^{(i)}, W_K^{(i)}, W_V^{(i)}$. After all heads finish, their outputs (each of dim 128) are concatenated back to a 4096-vector. Then a single learned matrix $W_O$ projects this back to dimension $d_{\text{model}}$. That $W_O$ is what HuggingFace calls o_proj — that's where the "o_proj" in target_modules comes from.
Why split the dimension across heads instead of running $h$ full-dimensional attentions? Cost. $h$ full attentions would be $h$ times as expensive. Splitting keeps total cost the same as a single big attention but lets different heads specialize. Empirically, different heads do learn different things: one might track subject-verb agreement, another tracks pronoun antecedents, etc.
If $h = 1$ (just one head, full-dimensional): you lose the ability to attend to different "kinds" of context in parallel. Empirically performs worse than multi-head at the same parameter count. If $h$ is too large (each head too thin, $d_k < 32$ or so): each head loses expressivity. The standard $d_k = 64$ or $128$ is a sweet spot.

2.10 The causal mask — why the model can't peek at the future

If you're predicting the next token, you can't be allowed to look at it during training. So before softmax we add a mask: $S'_{ij} = -\infty$ for all $j > i$. After softmax, $A_{ij} = 0$ for future positions.

Without the causal mask, the model would learn to cheat: predict token $t$ by simply attending heavily to token $t$ itself (which is what it's trying to predict). Useless model. The mask forces it to actually predict from context.

2.11 Quick worked example

Suppose $n=3$ tokens and $d_k = 2$. After computing $Q, K, V$:

Q = [[1, 0], [0, 1], [1, 1]]      # 3 query vectors
K = [[1, 0], [0, 1], [1, 1]]      # 3 key vectors
V = [[10, 0], [0, 10], [5, 5]]    # 3 value vectors

QK^T  = [[1, 0, 1],
         [0, 1, 1],
         [1, 1, 2]]

QK^T / sqrt(2)  =  ~[[0.71, 0, 0.71],
                     [0, 0.71, 0.71],
                     [0.71, 0.71, 1.41]]

After softmax per row (ignoring causal mask for clarity):
A ≈ [[0.42, 0.20, 0.38],
     [0.20, 0.42, 0.38],
     [0.21, 0.21, 0.58]]

Output = A @ V:
row 0:  0.42*[10,0] + 0.20*[0,10] + 0.38*[5,5] = [6.1, 3.9]
row 1:  0.20*[10,0] + 0.42*[0,10] + 0.38*[5,5] = [3.9, 6.1]
row 2:  0.21*[10,0] + 0.21*[0,10] + 0.58*[5,5] = [5.0, 5.0]

You can see token 3 (whose Q matches both K's strongly) ended up with a balanced mix; token 1's output is pulled toward V's that matched its Q most.

Chapter 3

3. The full transformer block — and why every piece is there

A transformer block is two sublayers: attention, then a feed-forward network (MLP). Each has a residual connection and a normalization. Let's go piece by piece.

3.1 The structure

# Pre-norm transformer block (modern style)
x = x + Attention(LayerNorm(x))
x = x + MLP(LayerNorm(x))

3.2 Residual connections (the "+ x" parts)

A deep network has many layers. If each layer fully replaces its input, signal and gradient have to survive the chain. Deep networks without skip connections train poorly — gradients vanish or explode. The residual ($y = f(x) + x$) lets the gradient flow through the $+x$ path no matter how messy $f$'s gradient is. It also lets each layer learn a small correction rather than an entire transformation, which is easier.
Remove residuals and a 32-layer transformer trains terribly (or not at all). Every modern architecture, vision or language, uses residual connections.

3.3 Layer normalization (the "LayerNorm" / "RMSNorm")

Inside each block, before the operation, normalize each token vector to have mean 0 and unit variance (LayerNorm) or just unit RMS (RMSNorm — Llama's choice):

$$\text{LayerNorm}(x) = \frac{x - \mu(x)}{\sigma(x)} \cdot \gamma + \beta$$
$\mu$ and $\sigma$ are computed per token vector. $\gamma, \beta$ are learned per-dimension scaling and shifting. Outputs are well-behaved regardless of the magnitude of inputs.
Without normalization, the variance of activations can blow up or vanish as you go deeper through layers. Gradients become unstable. LayerNorm enforces a clean signal scale at every layer. It also makes training robust to learning rate choice.
Llama uses RMSNorm (no mean subtraction, no $\beta$). It's slightly cheaper and works equally well. The difference matters at scale; at our level it's the same idea.

3.4 The MLP (feed-forward network) — and why SwiGLU

After attention, every token is sent through a small neural net independently. The classical transformer used:

$$\text{MLP}(x) = \text{ReLU}(x W_1) W_2$$

Modern LLMs use SwiGLU:

$$\text{SwiGLU}(x) = \big(\text{SiLU}(x W_{\text{gate}}) \odot (x W_{\text{up}})\big) W_{\text{down}}$$
Three matrices instead of two. $W_{\text{gate}}$ and $W_{\text{up}}$ both project up from $d_{\text{model}}$ (e.g., 4096) to a wider hidden dimension (e.g., 11008). One branch goes through SiLU (a smooth ReLU-ish activation), the other doesn't. They're multiplied element-wise ($\odot$). $W_{\text{down}}$ projects back to $d_{\text{model}}$.
Why three matrices instead of two? The gating mechanism — multiplying two branches — gives the MLP a form of multiplicative interaction, which is more expressive than pure addition. The SiLU-gated branch lets the model learn "let this through if relevant, suppress if not" per dimension. Empirically, SwiGLU outperforms ReLU MLPs at the same compute. PaLM showed this first; Llama adopted it.
Why is the hidden dimension wider than $d_{\text{model}}$? The MLP is where most "knowledge storage" in a transformer happens (research on this is ongoing, but neuron-storage-of-facts is well documented). Wider = more room to memorize patterns. Llama uses roughly $\frac{8}{3} d_{\text{model}}$ as the hidden dimension for SwiGLU.

The three matrices gate_proj, up_proj, down_proj are exactly what completes your seven-name target_modules list.

3.5 Stacking blocks

An LLM stacks 32 (7B), 80 (70B), or 128+ such blocks. The same operations apply at each layer; later layers see increasingly abstracted representations.

Chapter 4

4. Fine-tuning — what changes, what doesn't

4.1 The unit operation: a gradient step

Fine-tuning is the same algorithm as pretraining (gradient descent on cross-entropy), with three differences:

  • Starting point: pretrained weights, not random.
  • Data: your task-specific dataset, much smaller.
  • Steps: thousands, not trillions.

4.2 Why does fine-tuning work at all?

A pretrained model has already learned the building blocks of language: grammar, world knowledge, reasoning fragments. Your task usually requires a particular arrangement or emphasis of these, not new building blocks. So a few thousand gradient steps suffice — you're nudging the existing knob positions a small amount, not finding them from scratch.

4.3 Why we worry about catastrophic forgetting

If you fine-tune aggressively on a narrow dataset, the model can forget general skills it had. The knobs drift toward optima for your dataset, leaving the original optima. This is "catastrophic forgetting."

If you fine-tune Llama on just legal contracts for 10,000 steps at high LR: you might get a model that's great on contracts but suddenly bad at general chat. Mitigations: use low LR, train for few epochs, mix in some general data, or use PEFT (LoRA) which by limiting capacity preserves more of the base.

4.4 Full fine-tuning vs PEFT — the memory math

For a 7B model in mixed-precision (bf16) training with AdamW:

  • Model weights: 7B × 2 bytes (bf16) = 14 GB
  • Gradients: 7B × 2 bytes = 14 GB
  • Optimizer states (AdamW has 2 moments per parameter, fp32): 7B × 8 bytes = 56 GB
  • Activations (forward pass cache for backprop): ~10-20 GB depending on seq length
  • Total: ~95 GB for a 7B model in full fine-tuning

That's a multi-GPU setup. LoRA changes this picture dramatically.

Chapter 5

5. LoRA — every symbol explained

5.1 The setup again

You have a pretrained weight matrix $W \in \mathbb{R}^{d \times k}$. (For attention's q_proj in Llama-7B, $W$ is $4096 \times 4096$ — about 16.8 million parameters.) Fine-tuning would change it to $W + \Delta W$. LoRA's bet: $\Delta W$ has low rank.

5.2 What does "low rank" actually mean?

The rank of a matrix is the number of linearly independent directions of variation it contains. A rank-1 matrix is a single outer product $\mathbf{b}\mathbf{a}^\top$ — every column is a scalar multiple of $\mathbf{b}$. A rank-$r$ matrix can be written as a sum of $r$ such outer products, equivalently as a product of two skinny matrices:

$$\Delta W_{d \times k} = B_{d \times r} \cdot A_{r \times k} \quad \text{where } r \ll \min(d, k)$$
$A \in \mathbb{R}^{r \times k}$ projects from $k$-dim input down to $r$-dim "bottleneck." $B \in \mathbb{R}^{d \times r}$ projects back up from the $r$-dim bottleneck to $d$-dim output. Together, $BA$ is a $d \times k$ matrix that can only ever have rank at most $r$.
Why low-rank? Empirical observation: when you fine-tune a model fully and look at how much $\Delta W$ actually changes, it turns out to live in a very low-dimensional subspace. You don't need 16.8M new parameters per matrix — you need maybe a few hundred or a few thousand effective directions. Forcing $\Delta W = BA$ with small $r$ doesn't lose much expressivity in practice, and saves enormous parameters and memory.

5.3 The LoRA forward equation

$$h = Wx + \frac{\alpha}{r} BA\, x$$
$x$ is the input vector (e.g., 4096-dim). $W$ is the frozen pretrained matrix. $A, B$ are the trainable LoRA factors. $\alpha$ (alpha) is a scalar hyperparameter — the "amplification" applied to the LoRA branch. $r$ is the rank you chose.

5.4 Why initialize $B = 0$ but $A$ randomly?

At the very first forward pass, you want the model to behave exactly like the pretrained model — otherwise the loss might immediately spike and the first gradients are misleading. With $B = 0$, the LoRA contribution $BAx = 0$, regardless of $A$. So training starts from the pretrained model exactly. As soon as gradients flow, $B$ starts moving away from zero, and the adapter kicks in. If we initialized both $A$ and $B$ randomly, the model would start in some random vicinity of the base — bad for stability.
If we initialized both to zero: gradients of $A$ depend on $B$ and vice versa via the chain rule. Both being zero would freeze training of both. The "random $A$, zero $B$" asymmetry breaks the symmetry while still starting at $\Delta W = 0$.

5.5 The $\alpha / r$ scaling — and what happens if you change it

Without the scaling, as you increase $r$, the magnitude of $BA$ grows too (more terms in its sum). That means changing $r$ implicitly changes how big each gradient update is, forcing you to re-tune learning rate every time you change $r$. With the $\alpha/r$ scaling, you can change $r$ from 8 to 64 without re-tuning the learning rate. $\alpha$ becomes the "effective learning rate scale on the LoRA branch."
If $\alpha$ is too small (say $\alpha = 1, r = 64$): the LoRA branch is heavily downscaled, and adapter updates barely affect the model. Training will be slow, final quality limited. If $\alpha$ is too large (say $\alpha = 256, r = 8$): updates are amplified 32×, can cause instability, gradient explosions, or rapid overfitting. Common practice: $\alpha = r$ (scale factor 1) or $\alpha = 2r$ (gentle 2× amplification). Sweep if you're not sure.

5.6 Choosing $r$ — what each value means

rEffectWhen to use
2 – 4Very small adapter. Sufficient for very narrow tasks (style, format).Cheapest experiments.
8Default in most tutorials. Works for many tasks.First try.
16 – 32More capacity. Often best for substantive tasks.Reasoning, code, domain-specific instruct.
64 – 128High capacity. Diminishing returns above this.QLoRA paper used $r=64$. Heavy domain shift.

What if $r = 1$? Each $\Delta W$ can only encode one "direction" of update. Surprisingly, this works for very narrow tasks (sentiment, style) but breaks down on anything complex. What if $r$ equals the full rank? Then LoRA has the same expressivity as full fine-tuning. But you pay the same memory cost, and there's no benefit over just training $W$ directly.

5.7 LoRA dropout — what it is and what changing it does

LoRA optionally applies dropout to the input of the LoRA branch:

$$h = Wx + \frac{\alpha}{r} BA \cdot \text{Dropout}(x, p)$$
With probability $p$, each entry of $x$ entering the LoRA branch is zeroed out (and the rest scaled to compensate). The frozen branch $Wx$ is untouched.
Dropout is a regularizer. With small fine-tuning datasets and an expressive adapter, the model can overfit (memorize the train set, generalize poorly). Dropout makes the adapter robust to missing input signals, reducing overfitting.
$p = 0$: no regularization, adapter is fully responsive. Good if you have lots of data or aren't overfitting. $p = 0.05$ to $0.1$: mild regularization, standard for small fine-tuning datasets. $p > 0.2$: aggressive, can hurt learning by removing too much signal. Tune if you see train loss going down but eval loss going up (the textbook overfit signature).

5.8 Choosing target_modules — what each combination buys

ChoiceTrainable paramsQuality
["q_proj", "v_proj"] only~0.06% of baseOriginal LoRA paper. Surprisingly OK; cheapest.
All 4 attention ["q_proj","k_proj","v_proj","o_proj"]~0.12%Better at relational tasks. Cheap.
All 7 (attention + MLP)~0.4%Best quality, modest extra cost. The current default.
Why MLP matters: MLPs hold most of the "knowledge" of a transformer (factual associations, learned patterns). If your fine-tuning task involves new factual knowledge or new skill patterns, MLP-LoRA helps a lot. If it's just style or format, attention-only LoRA might be enough.
Empirical finding from QLoRA and follow-ups: more modules with smaller $r$ > fewer modules with bigger $r$ at equal parameter budget. So if you have a budget for, say, 1M trainable params: better to do $r = 8$ on all 7 modules than $r = 64$ on just Q,V.
Chapter 6

6. QLoRA — what 4-bit quantization actually means

6.1 What is a quantized number?

A 32-bit float (fp32) can represent ~$2^{32}$ ≈ 4 billion distinct values, with high precision. A bf16 represents ~65,000 distinct values, with lower precision (but the same exponent range as fp32, which is why it's preferred for training). A 4-bit number can represent only $2^4 = 16$ distinct values.

"Quantization" means: take a high-precision number and snap it to the nearest of 16 (or 256, for 8-bit) representable values. You lose information, but use far less memory.

6.2 Block-wise quantization with a scale

You can't quantize a weight in isolation if you only have 16 codes — there's no fixed mapping that works for all magnitudes. So you do it in blocks:

  1. Take a block of 64 weights.
  2. Find their absolute max, call it $s$ (the "scale").
  3. Map each weight $w \in [-s, s]$ to the nearest of 16 levels by computing $\text{round}(w / s \cdot 7.5)$ — gives an integer in $[-8, 7]$.
  4. Store: the integer (4 bits) and the scale $s$ for the block (stored once per 64 weights).

When reading: $\hat{w} = \text{integer} \cdot s / 7.5$. Lossy but compact.

6.3 Why "NF4" — normal float 4

QLoRA's clever observation: a pretrained model's weights are roughly normally distributed, centered at zero. So most weights are near zero, with rare weights far from it. Evenly-spaced 4-bit codes waste precision on regions where there are few weights.

NF4 spaces the 16 codes according to the quantiles of a standard normal distribution. Code values cluster densely near zero (where most weights are) and spread out in the tails. This puts representable values where actual weights are, minimizing error.
With uniform 4-bit (16 evenly-spaced codes from $-s$ to $s$): a weight close to zero might get snapped to the nearest code, which could be relatively far away. With NF4: weights near zero have a code right next to them. NF4 produces measurably lower quantization error than int4 on normally-distributed weights.

6.4 Double quantization — quantizing the scales

Each block of 64 weights has its own fp32 scale. With billions of weights, those scales themselves take up nontrivial memory:

  • 7B weights / 64 per block = 110M blocks
  • 110M scales × 4 bytes (fp32) = 440 MB just in scales

Double-quantization: quantize the scales themselves to 8-bit. Saves ~0.4 bits per weight on average.

6.5 Paged optimizers — what they do

During backward pass and the AdamW update, optimizer states have momentary spikes. Even when training "fits" in memory most of the time, these spikes can OOM the GPU.

QLoRA uses NVIDIA's unified memory: optimizer states live in CPU RAM by default. When the GPU needs them, they're paged in like virtual memory. The GPU never holds the full optimizer state at once. Trainability survives spikes that would otherwise crash. The cost is a small slowdown during paging, but the alternative is "doesn't fit at all."

6.6 The compute dtype — why bf16 even though weights are 4-bit

Weights are stored in 4-bit. But during the forward pass, when you actually compute $Wx$, you can't do matrix multiply in 4-bit precision — that would lose far too much. So:

  1. For each linear layer call, dequantize a block back to bf16.
  2. Compute $Wx$ in bf16.
  3. Drop the bf16 weights (don't store them long-term).
bnb_4bit_compute_dtype = bfloat16 means "when actually computing with these 4-bit weights, dequantize to bf16 for the math." Could be fp16, but bf16 has better numerical range and is preferred on modern GPUs.

7. RL fine-tuning — every term, every "why"

7.1 What is a policy?

In RL terminology, a policy $\pi_\theta(\cdot \mid s)$ is anything that, given a state, gives a probability distribution over actions. Your LLM is a policy. Given a prompt (state), it gives a distribution over next tokens (actions). Sampling from it is "rolling out" the policy.

7.2 What is a reward?

A reward is just a scalar number we assign to a completed trajectory (prompt + full response) that says "how good was this?" For math problems: $+1$ if the answer matches ground truth, $0$ otherwise. For code: $+1$ if it passes tests. For RLHF: a learned reward model that scores helpfulness.

Why is RL needed beyond supervised fine-tuning? Because for some tasks, you don't have a "correct answer" to imitate — you have a notion of "better or worse." A math problem has many valid solution paths leading to the same answer. SFT can only show one. RL can reward any path that reaches the right answer, letting the model discover its own.

7.3 The objective — what we're maximizing

$$\mathcal{J}(\pi_\theta) = \mathbb{E}_{\mathbf{q} \sim p_\mathcal{Q}}\!\Big[\mathbb{E}_{\mathbf{o} \sim \pi_\theta(\cdot \mid \mathbf{q})}[R(\mathbf{q}, \mathbf{o})] - \beta\, \mathbb{D}_{KL}\big[\pi_\theta(\cdot \mid \mathbf{q}) \| \pi_{\text{ref}}(\cdot \mid \mathbf{q})\big]\Big]$$
$\mathbf{q}$ is a prompt sampled from your dataset $p_\mathcal{Q}$. $\mathbf{o}$ is a response sampled from the current policy. $R$ is the reward, a scalar. $\mathbb{D}_{KL}$ is KL divergence — a measure of how different two probability distributions are. $\pi_{\text{ref}}$ is a fixed reference policy (usually the SFT model you started from). $\beta$ is a coefficient ≥ 0 controlling how much we penalize drifting from the reference.
The two terms balance: maximize reward, but don't run away too far from your starting model. Why care about staying near $\pi_{\text{ref}}$? Because the reward function is usually imperfect. If you let the policy maximize a learned reward model without restraint, it discovers exploits (high reward, terrible text) — this is called "reward hacking." The KL term is the leash.
$\beta = 0$ (no KL): policy is free to diverge arbitrarily. Fine for verifiable rewards (math, code) where exploitation isn't an issue. Used by R1-Zero and most reasoning-RL setups. $\beta = 0.1$: moderate penalty, common in RLHF. $\beta = 1$: very tight leash, model barely moves from reference.

7.4 The policy gradient theorem (the math behind RL updates)

We can't compute the gradient of an expectation directly. The "policy gradient theorem" gives us:

$$\nabla_\theta \mathbb{E}_{\mathbf{o} \sim \pi_\theta}[R(\mathbf{o})] = \mathbb{E}_{\mathbf{o} \sim \pi_\theta}\big[R(\mathbf{o}) \cdot \nabla_\theta \log \pi_\theta(\mathbf{o})\big]$$
Sample responses from the current policy, score them with $R$, and use that scalar to scale the gradient of the log-probability of producing that response. High-reward responses get gradient pushing the policy to produce them more often; low-reward ones get gradient pushing them less.
Why the log? Because $\nabla \pi / \pi = \nabla \log \pi$ — this trick (the "log derivative" or "REINFORCE" trick) turns an unwieldy gradient through sampling into a gradient through log-probabilities, which we can compute.

7.5 The variance problem — and why we need a baseline

The raw policy gradient has huge variance. Imagine all rewards are between 90 and 100. Subtracting 95 doesn't change which response is better — but the magnitudes of gradient updates become much smaller and more focused on differences.

The variance-reduced estimator uses an advantage instead of raw reward:

$$\nabla_\theta \mathcal{J} = \mathbb{E}\big[\hat{A} \cdot \nabla_\theta \log \pi_\theta(\mathbf{o})\big]$$ $$\text{where } \hat{A} = R - b \text{ for some baseline } b$$
$\hat{A}$ is the advantage — how much better than "average" this response was. As long as the baseline $b$ doesn't depend on the action taken, subtracting it doesn't change the expected gradient but greatly reduces variance.
With a baseline, a "decent" response gets near-zero gradient (it was about average, no strong signal). A great response gets large positive gradient (boost it). A bad response gets negative gradient (suppress it). This focuses learning on differences and is the foundation of all modern policy gradient methods.

7.6 PPO — what each piece does

PPO's objective per token:

$$\mathcal{J}_{\text{PPO}} = \mathbb{E}\left[\sum_t \min\!\Big(r_t(\theta)\hat{A}_t,\ \text{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon)\hat{A}_t\Big)\right]$$ $$r_t(\theta) = \frac{\pi_\theta(o_t \mid s_t)}{\pi_{\theta_{\text{old}}}(o_t \mid s_t)}$$
$r_t(\theta)$ is the importance ratio: how much more likely the new policy is to produce this token than the old policy. $\hat{A}_t$ is the advantage. The $\min$ and $\text{clip}$ together implement the "trust region" — don't let the policy move too far from where you last sampled.
Why the importance ratio? Because we sampled $\mathbf{o}$ from $\pi_{\theta_{\text{old}}}$, but we're updating $\pi_\theta$. The ratio re-weights the gradient to account for the mismatch. If $\pi_\theta$ has drifted far from $\pi_{\theta_{\text{old}}}$, the samples no longer represent $\pi_\theta$ well, and the gradient estimate becomes unreliable.
Why clip? If the importance ratio gets very large (the new policy puts way more weight on this token than the old), one bad estimate could give a huge gradient step. Clipping caps how much any single token can pull the policy. The $\min(\cdot, \text{clip}(\cdot))$ formula clips only when this would otherwise increase the objective unduly — it lets bad updates flow normally but caps good updates.
$\epsilon = 0.2$ is the standard. $\epsilon = 0.05$ (tight): the policy barely moves per update; you'll need many more updates. $\epsilon = 0.5$ (loose): policy can lurch wildly per step; instability returns. The value 0.2 is empirical and has held up across many RL applications.

7.7 What's a "value function" and why GRPO ditches it

To compute $\hat{A}_t$ — the advantage at each token — PPO uses a learned value function $V_\phi(s_t)$ that estimates "expected reward from this state onwards." Then $\hat{A}_t \approx R - V_\phi(s_t)$ (the actual return minus what we expected).

$V_\phi$ is another neural network with its own parameters $\phi$, the same size as the policy, trained alongside it. For a 70B policy, you have a 70B critic — your memory just doubled.
GRPO realized: for verifiable-reward tasks, instead of a learned baseline, sample $G$ responses to the same prompt and use the group mean as the baseline. No critic needed.

7.8 GRPO step by step

For each prompt $\mathbf{q}$:

  1. Sample $G$ responses $\{\mathbf{o}_1, \ldots, \mathbf{o}_G\}$ from $\pi_{\theta_{\text{old}}}$.
  2. Score each response with the reward function: $R_i = R(\mathbf{q}, \mathbf{o}_i)$.
  3. Compute group-normalized advantage for response $i$:
$$\hat{A}_i = \frac{R_i - \text{mean}(R_1, \ldots, R_G)}{\text{std}(R_1, \ldots, R_G)}$$
$\hat{A}_i$ is a single scalar per response, used as the advantage for every token in that response. Positive means "this response was better than the average of the $G$"; negative means worse. Standardizing (subtract mean, divide by std) gives a clean zero-mean unit-variance signal.
Why divide by std? To normalize the magnitudes — without it, a prompt with rewards ranging from 0.4 to 0.6 produces tiny advantages, while one with rewards 0.0 to 1.0 produces large advantages. The std-normalization makes them comparable across prompts of different difficulty.

7.9 Dr.GRPO — the bugs in the above, and the fix

The Dr.GRPO paper noticed the std-normalization actually introduces bias rather than removing it. Here's why:

Bug 1: Response-length bias (the $\frac{1}{|\mathbf{o}_i|}$ in the loss)

In GRPO's full loss, each response's per-token loss is summed and then divided by its length: $\frac{1}{|\mathbf{o}_i|} \sum_t \ldots$. Combined with $\hat{A}_i$ being shared across all tokens of that response, this means: a response of 1000 tokens contributes only $\frac{1}{1000}$ per token. A response of 100 tokens contributes $\frac{1}{100}$ per token.

Why is this a bug? Imagine a correct response (positive $\hat{A}$): if it's short, each token gets a large positive gradient — the model learns "produce these short correct answers." If it's long, each token gets only a small positive gradient. So the model favors brevity on correct answers. Conversely, for incorrect responses (negative $\hat{A}$): a long wrong response has each token penalized only weakly. The model learns it's "safer to be long when wrong." Over training, this produces: correct answers get shorter, wrong answers get progressively longer. The famous "RL response length keeps growing" is partly artifact, partly real reasoning improvement.

Bug 2: Question-difficulty bias (the $\text{std}(R)$ in the denominator)

For an easy question, almost all $G$ responses are correct — rewards cluster, low std → large advantages after dividing. Same for impossible questions: almost all wrong → low std → large negative advantages. For medium-difficulty questions (the most informative!), std is high → advantages are small. The model spends gradient on easy and impossible questions and neglects the medium ones, which is exactly backwards.

The Dr.GRPO fix

$$\hat{A}_i^{\text{Dr.GRPO}} = R_i - \text{mean}(R_1, \ldots, R_G)$$

And in the loss, sum over all tokens of all responses with a constant normalizer (max length) instead of per-response length normalization.

Remove std-normalization → eliminate difficulty bias. Remove per-response length-normalization → eliminate length bias. The result: responses don't bloat on wrong answers, and medium-difficulty problems get proper gradient attention. Token efficiency goes up, and benchmark scores at fixed training time go up.

7.10 DAPO — what each of the four tricks fixes

DAPO is GRPO with four practical surgeries:

Trick 1: Clip-Higher (asymmetric clipping)

Instead of clipping to $[1-\epsilon, 1+\epsilon]$, clip to $[1-\epsilon_{\text{low}}, 1+\epsilon_{\text{high}}]$ with $\epsilon_{\text{high}} > \epsilon_{\text{low}}$.

Symmetric clipping treats both directions equally, but their effects differ. A "high-probability-decrease" clip (lower bound) prevents over-suppressing tokens, which is fine. A "high-probability-increase" clip (upper bound) prevents the policy from confidently increasing the probability of promising-but-rare tokens — which kills exploration. Loosening the upper clip lets the policy boost low-probability tokens more aggressively when they earn high reward, preserving diversity.
Typical: $\epsilon_{\text{low}} = 0.2$, $\epsilon_{\text{high}} = 0.28$. Modest asymmetry, big effect on exploration.

Trick 2: Dynamic Sampling

If all $G$ responses to a prompt are correct (or all wrong), the group mean equals each $R_i$, so every advantage is zero. That prompt contributes no gradient to that step. Wasted compute.

DAPO oversamples prompts, then keeps only those whose groups have at least one correct and one incorrect response — i.e., informative groups. The batch is filled with prompts where the model is on the boundary of competence (the most informative regime).
Without dynamic sampling, late in training when the model has solved most easy problems, more and more prompts produce all-correct groups, and effective gradient per step plummets. Dynamic sampling keeps the gradient pipeline full.

Trick 3: Token-level policy gradient loss

Same as Dr.GRPO's fix for the length bias: instead of averaging losses per-response then averaging across responses (which gives short responses disproportionate weight), sum token losses over the entire batch and divide by total token count.

Trick 4: Overlong reward shaping

If a response exceeds a soft cap (e.g., max_length - 4096), apply a length-proportional negative shaping. Beyond the hard cap, truncate.

Without this, when the model is wrong it can generate runaway 32K-token chains. These are computationally expensive and add noise to the reward signal (a partially-correct chain gets reward 0 even if it was on the right track). Soft-shaping discourages the model from going past where it's productive.

7.11 DRA-GRPO — why diversity matters

Vanilla GRPO only cares about which response got which reward. It doesn't notice if 7 of 8 correct responses are essentially the same proof. The policy then collapses onto one strategy, losing the diversity needed to discover novel approaches later.

DRA-GRPO measures semantic similarity between the $G$ responses (e.g., via a sentence embedding model), then computes a redundancy penalty. Responses that are unique among the correct ones get a small reward boost; responses that are duplicates of others get downweighted.
Maintaining multiple distinct solution strategies during training keeps the policy distribution wide enough to discover combinations and generalizations. Mode collapse — converging on one strategy — is a known failure mode of RL on language tasks, especially with small $G$.

7.12 λ-GRPO — the credit assignment problem

In a 2000-token chain of reasoning ending in a correct answer, which tokens caused the success? Vanilla GRPO doesn't know — it assigns the same advantage to every token. Filler words and the actual insight get equal credit.

Classical RL solves this with a value function and TD-learning, but GRPO threw away the value function. λ-GRPO restores partial credit assignment without a critic:

λ-GRPO uses eligibility traces: when a token earns reward at the end of a response, it propagates credit backward through prior tokens with a decay factor $\lambda \in [0,1]$. Tokens near the reward get most credit; tokens far back get less. Implemented via token log-probabilities as a critic-free proxy for TD error.
$\lambda = 0$: all credit to immediate tokens (very local, high bias). $\lambda = 1$: equal credit to all tokens (back to GRPO, high variance). Intermediate $\lambda$ (e.g., 0.95) blends both, getting most of the benefit of both extremes. Reported gains: 30-40% better training curves, +3-4.5 points absolute on benchmarks.

8. Hyperparameters — what each one means and what changes when you turn it

Every hyperparameter is a dial. Let's go through them, one by one, with what they control, default values, and the actual symptoms you'll see if they're set wrong.

9. Putting it together — a worked example end to end

Let's walk through fine-tuning a 7B model to be better at solving math, using QLoRA + GRPO. This is roughly the recipe DeepSeek-R1-Zero used (at much larger scale).

9.1 Setup

Base model: Qwen2.5-Math-7B
Goal: improve accuracy on AIME-style math problems
Hardware: 1× A100 80GB

# Load base model in 4-bit
from transformers import AutoModelForCausalLM, BitsAndBytesConfig
import torch

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_use_double_quant=True,
    bnb_4bit_compute_dtype=torch.bfloat16,
)

model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-Math-7B",
    quantization_config=bnb_config,
    device_map="auto",
)

# Attach LoRA adapters to all 7 linear modules
from peft import LoraConfig, get_peft_model

lora_config = LoraConfig(
    r=32,
    lora_alpha=64,           # 2r
    lora_dropout=0.05,
    target_modules=["q_proj","k_proj","v_proj","o_proj",
                    "gate_proj","up_proj","down_proj"],
    bias="none",
    task_type="CAUSAL_LM",
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: ~ 80M (1.0% of base)

9.2 What's happening under the hood

  • Base 7B weights are stored in NF4. ~3.5 GB on GPU instead of ~14 GB.
  • LoRA adapters in bf16: ~80M parameters × 2 bytes = 160 MB.
  • Gradients are computed only for LoRA params: 160 MB more.
  • Paged AdamW keeps optimizer state in CPU mostly: minimal GPU overhead.
  • Total GPU memory: roughly 6-8 GB at idle, peak around 30-40 GB during long rollouts.

9.3 GRPO training loop (conceptually)

for step in range(num_steps):
    prompts = sample_batch_of_math_problems(batch_size=64)

    # Step 1: Generate G responses per prompt (sample rollouts)
    all_responses = []
    for q in prompts:
        responses = generate(model, q, num_return_sequences=G, temperature=0.7)
        all_responses.append(responses)

    # Step 2: Score each response with rule-based verifier
    all_rewards = []
    for q, responses in zip(prompts, all_responses):
        rewards = [1.0 if verify(r, ground_truth[q]) else 0.0
                   for r in responses]
        all_rewards.append(rewards)

    # Step 3: Compute advantages (Dr.GRPO style — no std normalization)
    advantages = []
    for rewards in all_rewards:
        mean_r = sum(rewards) / len(rewards)
        advs = [r - mean_r for r in rewards]   # Dr.GRPO
        advantages.append(advs)

    # Step 4: PPO-style update with clipping
    for ppo_epoch in range(4):
        for q, responses, advs in zip(prompts, all_responses, advantages):
            for response, adv in zip(responses, advs):
                # Compute new log-probs of the same tokens
                logp_new = compute_logprobs(model, q, response)
                logp_old = stored_logprobs[q, response]

                ratio = exp(logp_new - logp_old)
                loss_a = ratio * adv
                loss_b = clip(ratio, 1-0.2, 1+0.2) * adv
                loss = -min(loss_a, loss_b).mean()

                loss.backward()
        optimizer.step()
        optimizer.zero_grad()

9.4 What you'd observe

  • Early steps: Most rollouts get reward 0 (model is bad at hard problems). Advantages mostly slightly negative. Slow improvement.
  • Around step 200-500: The first prompts start producing mixed rollouts (some correct, some not). Advantages have real signal. Model accuracy on eval starts climbing.
  • Mid-training: Response length grows (sign of more reasoning, or possibly Dr.GRPO bias — depends on your loss formulation).
  • Late training: Diminishing returns; accuracy plateaus. Some "Aha-moment" patterns emerge.

9.6 The full picture

Everything we've covered now connects:

  • The pretrained model is a transformer stack of attention + SwiGLU MLP blocks. The seven linear matrices per block are the LoRA targets.
  • QLoRA shrinks the frozen base via NF4 quantization with double-quantization and runs LoRA adapters in bf16 on top.
  • LoRA represents each weight update as $\frac{\alpha}{r} BA$, training only $A$ and $B$ — typically <1% of parameters.
  • SFT trains via cross-entropy on supervised examples.
  • RL fine-tuning trains via policy gradient on samples scored by a reward.
  • GRPO uses the group of $G$ samples per prompt as a baseline, eliminating the value model.
  • Dr.GRPO removes two biases (length normalization and std normalization) for unbiased optimization.
  • DAPO adds four practical tricks: clip-higher, dynamic sampling, token-level loss, overlong shaping.
  • DRA-GRPO adds diversity awareness to prevent mode collapse.
  • λ-GRPO adds critic-free credit assignment via eligibility traces.
  • Hyperparameters control the size, schedule, and stability of each gradient update — and each one has a specific failure mode if mis-set.

If you walked through this whole document, you now know not just what the formulas say, but why every term is there and what would happen if you took it out. That's enough foundation to read any modern fine-tuning paper without getting lost in notation.
